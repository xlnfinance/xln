# Code Cleanup Plan

Last review: 2026-05-29

This plan is intentionally conservative. Delete or move code only when the proof
is current. Age, filename, and old planning notes are not proof.

The active backlog is the root [todo.md](../todo.md). This file is only the
cleanup/readability policy for code and docs.

## Completed Cleanup

- Stale live planning files were consolidated into [todo.md](../todo.md).
- The broken root `next.md` symlink was removed.
- Duplicate frontend static TODO/NEXT pages were removed so stale planning
  snapshots are not served as current docs.

  only active branch.

## Safe Readability Work

These are safe directions because they do not require protocol behavior changes.
They should still be done in small patches with targeted tests.

### Runtime

1. Keep the current top-level split and document boundaries in
   [runtime/README.md](../runtime/README.md).
2. Keep `relay`, `routing`, `orderbook`, `networking`, `orchestrator`,
   `scripts`, `account-tx`, and `entity-tx` as the canonical buckets.
3. Deduplicate recovery/watchtower normalizers only when tests cover the exact
   before/after behavior:
   - lookup key normalization;
   - tower mode normalization;
   - non-negative integer clamping;
   - bundle/receipt sort comparators.
4. Split persistence/replay internals out of `runtime/runtime.ts` only after a
   no-behavior-change plan is proven by persistence and E2E gates.

### Frontend

1. Keep `components`, `stores`, `utils`, `view`, `network3d`, and `types` as the
   canonical frontend buckets.
2. Prefer extracting payment/recovery helpers from large Svelte components over
   moving route-level files.
3. Centralize selector refresh logic only after tests prove no lost runtime,
   entity, or account selection state.

### Docs

1. Keep one live backlog: [todo.md](../todo.md).
2. Keep main protocol descriptions in `docs/`.
3. Keep generated/static docs free of stale TODO/NEXT snapshots.
4. Leave `docs/archive/` as historical context, not active planning.

## Not Safe To Remove Blindly

- anything under `runtime/scenarios/` except obvious OS junk;
- anything in `frontend/src/lib/view/`;
- anything in `frontend/src/lib/network3d/`;
- anything mentioned only by docs until import and route usage are both checked;
- archived docs unless the user explicitly asks to remove history.

## E2E Standard To Keep Applying

1. The first comment block in each E2E spec should state the user flow and goal.
2. Primary assertions should combine saved runtime/event truth with visible HTML
   confirmation when applicable.
3. Prefer shared helpers over file-local duplicates.
4. Use `expect.poll` for node-side polling where possible.
5. Keep isolated-browser coverage for:
   - payment;
   - swap;
   - custody deposit/withdraw;
   - wiped-browser watchtower recovery;
   - post-restore channel payments;
   - dispute and delayed-last-resort watchtower behavior.
